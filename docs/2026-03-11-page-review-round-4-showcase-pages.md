# Deskhand V2 Page Review Round 4

> 日期：2026-03-11
> 范围：第 3 页 `generative-ui-solution`、第 6 页 `skill-system-features`、第 9 页 `clipboard-features`
> 背景：这三页当前都使用双图 showcase 模板，但展示效果仍然不理想
> 用途：交给实现窗口继续迭代

## 1. 这轮 review 聚焦什么

这轮不讨论 header、不讨论架构页，也不讨论文案微调。

只讨论一类页面：

**双图 showcase 页。**

对应三页：

1. `generative-ui-solution`
2. `skill-system-features`
3. `clipboard-features`

用户反馈的核心感受是：

- 现在像先画了两个小格子，再把截图塞进去
- 截图本来应该是主角，但现在缩在两个小窗口里
- 打开页面后，第一眼看到的是“卡片容器”，不是“产品画面”

一句话判断：

**这三页现在被做成了“截图卡片页”，但正确方向应该是“截图主导页”。**

---

## 2. 这轮已经确认的主问题

用户已明确选择的主要问题：

1. `窗口套窗口感太重`
2. `图片面积优先级不够`
3. `截图和页面风格不融合`
4. `图像容器太像缩略图，不像展示图`
5. `这一类页面被当成了信息卡片页`

可选问题：

1. `两张图被平均缩小`
2. `两张图之间关系表达不清`

这两个不是先验结论，而是看最终视觉方案决定是否需要进一步处理。

---

## 3. 根因判断

当前这三页的问题，不是单张截图有问题，而是模板的默认心智错了。

当前实现大致是：

1. 页面先渲染标题区
2. 再渲染双列 grid
3. 每张图都包在一个 `ScreenCard` 里
4. `ScreenCard` 再包含：
   - 外层 card
   - 内层 frame
   - 图片本体
   - 下方 caption

这会产生几个结果：

### 根因 A：卡片感压过了截图本身

双层边框、圆角、阴影、内边距，会让截图更像被塞进组件里的内容，而不是整页主视觉。

### 根因 B：截图面积被系统性压缩

标题、lead、外层 card padding、内层 frame padding、caption 都在吃空间。

最后留给图像本体的面积太少。

### 根因 C：页面类型判断错了

这类页的本质不是“信息卡片页”，而更接近：

- 作品证据页
- 产品陈列页
- 画册式双图页

也就是说：

**图应该先说话，文案退后。**

### 根因 D：视觉语言不一致

整站是 warm editorial / paper-like 的方向，但这三页的截图容器更像 dashboard 小部件。

所以截图和页面外壳之间会有明显断层，产生“像被丢进去”的感觉。

---

## 4. 第四轮总方向

这三页不建议继续沿用当前 `ScreenCard` 思路。

推荐新增一类明确的 showcase 版式：

**`editorialShowcase`**

它的核心原则是：

1. 截图是主角
2. 容器感降到最低
3. 标题区变薄
4. caption 缩成轻标签，而不是完整说明卡
5. 双图按原始比例分配空间，不强求机械对称

这里有个关键判断：

**同等重要，不等于必须等宽等高。**

如果两张图原始比例差很多，硬做成两个一样的盒子，通常只会让两张图一起变小。

---

## 5. 推荐的统一版式家族

推荐这三页统一进入同一家族：

**画册式双图页**

基础结构：

```text
+--------------------------------------------------------------+
| kicker / title / lead（薄）                                  |
+----------------------------------------+---------------------+
|                                        |                     |
|               图 A 主视觉               |      图 B 主视觉     |
|                                        |                     |
|                                        |                     |
+----------------------------------------+---------------------+
| A label + 1 句说明                      | B label + 1 句说明   |
+--------------------------------------------------------------+
```

这套家族的意图不是“做两个卡片”，而是：

- 一页同时陈列两张证据
- 图像本体尽可能大
- 说明文字只做最轻的引导

---

## 6. 第 3 页：`generative-ui-solution`

### 当前页面关系

这一页两张图原始比例差异比较明显：

- 左图 `Playground` 更宽，更像完整工作台
- 右图 `This or That` 更窄，更像偏好测试面板

它们都重要，但不适合做成两个同尺寸窗口。

### 推荐方案

采用画册式双联页，但明确按原始比例分配空间：

```text
+--------------------------------------------------------------+
| Generative UI / Page 2                                       |
| 把探索过程变成可见、可比较、可收敛的交互流                    |
| lead（压薄）                                                 |
+-------------------------------------------+------------------+
|                                           |                  |
|                                           |                  |
|             Playground 大图                |   This or That   |
|           作为更宽的主视觉                 |      窄图         |
|                                           |                  |
|                                           |                  |
+-------------------------------------------+------------------+
| Playground / 1句                           | This or That / 1句|
+--------------------------------------------------------------+
```

### 设计原则

1. 左图更宽，但不意味着右图是配角
2. 两张图都不再使用厚边框卡片
3. caption 退化为轻标签 + 一句说明

### 目标

让用户一打开这页，就直接看到：

- 左边是完整工作台
- 右边是偏好测试机制

而不是先看到两个“组件盒子”。

## 7. 第 6 页：`skill-system-features`

### 当前页面关系

这一页的两张图虽然也有主次差异，但没有第 3 页那么强。

更重要的是：

- 左图是行为洞察与推荐闭环
- 右图是能力池管理

这两张图在论证上是一组互补关系。

### 推荐方案

同样使用画册式双联页，但左右差异比第 3 页更克制：

```text
+--------------------------------------------------------------+
| Skill System / Page 2                                        |
| 把 skill 的发现、寻找、安装，全部交给 Agent                  |
| lead（压薄）                                                 |
+----------------------------------------+---------------------+
|                                        |                     |
|                                        |                     |
|            Skill Insight                |    Skill Pool       |
|         更大一些，但不要压太狠          |    略窄一些         |
|                                        |                     |
|                                        |                     |
+----------------------------------------+---------------------+
| Skill Insight / 1句                     | Skill 能力池 / 1句  |
+--------------------------------------------------------------+
```

### 设计原则

1. 保留左右双图的对应关系
2. 左图可以略大，但不要夸张到像主从结构
3. 把“能力池”从小窗口感里解放出来，不再像右侧附属面板

### 目标

这一页要呈现的是：

- 左边是 Agent 如何发现并推荐
- 右边是这些能力最终如何沉淀与管理

所以要让两张图读起来像一组系统能力，而不是“一大一小两个截图卡”。

---

## 8. 第 9 页：`clipboard-features`

### 当前页面关系

这一页和前两页不一样。

它的两张图在叙事上更接近：

- 左图：工作轨迹回放
- 右图：进一步洞察与总结

所以这页可以比前两页更强调“前后递进”关系。

### 推荐方案

第 9 页优先尝试两种方案中的第一种：

#### 第一推荐：仍然使用画册式双联页

```text
+--------------------------------------------------------------+
| Clipboard / Page 2                                           |
| AI 可以主动读取剪贴板历史，作为上下文来完成任务              |
| lead（压薄）                                                 |
+----------------------------------------+---------------------+
|                                        |                     |
|             Clipboard-1                 |    Clipboard-2      |
|           工作轨迹回放                  |   特征 / 时间分布    |
|                                        |                     |
+----------------------------------------+---------------------+
| Clipboard-1 / 1句                       | Clipboard-2 / 1句    |
+--------------------------------------------------------------+
```

#### 第二推荐：轻微递进式布局

如果第一种做完仍然显得太“平均”，再尝试：

```text
+--------------------------------------------------------------+
| title / lead                                                 |
+--------------------------------------------------------------+
|                        Clipboard-1                           |
|                       更宽、更完整                            |
+----------------------------------------------+---------------+
| Clipboard-1 / 1句                             | Clipboard-2   |
|                                              | 小一些但清楚  |
|                                              | 1句说明       |
+----------------------------------------------+---------------+
```

### 设计原则

1. 默认先不做过强主次
2. 如果平均双联仍然显得呆板，再考虑轻微递进
3. 不要让右图重新退化成一个小附图

### 目标

这页的关键不是单纯展示两张图，而是让用户感受到：

**Agent 不只拿到了历史，还做出了额外洞察。**

---

## 9. 这轮明确不建议做的事

1. 不继续沿用当前 `ScreenCard` 的双层容器感
2. 不给每张图保留厚重卡片边框
3. 不让 caption 继续占据明显高度
4. 不强行把所有双图页做成完全对称的两个格子
5. 不把这三页继续当作“卡片信息页”处理

---

## 10. 实现动作

### 组件层

1. 为 showcase 页面增加新的 `editorialShowcase` 家族
2. 支持“按比例分配双图宽度”，而不是只支持 `featured / balanced`
3. 支持轻标签式 caption，而不是默认完整卡片说明

### 样式层

1. 去掉双图页里厚重的 `ScreenCard` 外层容器感
2. 降低边框、阴影、内层 frame 的存在感
3. 让标题区更薄，把垂直空间优先留给图
4. 双图之间更像“陈列间距”，而不是“卡片间距”

### 内容层

1. 第 3 页按明显比例差异处理
2. 第 6 页按克制的不对称处理
3. 第 9 页先按双联处理，必要时再尝试轻递进

---

## 11. 验收标准

实现后应满足：

1. 用户打开第 3、6、9 页时，第一眼看到的是两张更大的截图，而不是两个小窗口
2. 截图容器感明显下降，不再像“窗口套窗口”
3. 双图页和整站 warm editorial 风格更一致
4. caption 不再抢图的空间和注意力
5. 至少第 3 页和第 6 页，不再给人“两个主角一起缩进小格子里”的感觉
6. 第 9 页如果引入轻微递进关系，必须比当前更自然；如果效果不好，宁可保留双联，不强上
