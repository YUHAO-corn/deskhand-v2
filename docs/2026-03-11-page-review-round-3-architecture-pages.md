# Deskhand V2 Page Review Round 3

> 日期：2026-03-11
> 范围：第 7 页 `skill-system-architecture`、第 10 页 `clipboard-architecture`
> 背景：第二轮修复已执行，但这两页的架构图仍然没有达到“主证据完整可读”的目标
> 用途：交给实现窗口继续迭代

## 1. 这轮只处理一个问题

第二轮之后，第 7 页和第 10 页虽然已经换成了 `decision rail`，但效果仍然不理想。

用户新的反馈点很明确：

**两张架构图还是没有被足够完整、足够清晰地展示出来。**

这里的“没有展示完整”不一定是字面意义上的裁切，更准确地说，是：

- 图被压得太小
- 关键结构看不清
- 页面第一眼仍然不是“先看图，再读决策”

所以第三轮不再讨论 CTA、文案或其他页面，只聚焦这两页的版式问题。

---

## 2. 已确认的事实

### 图片尺寸

- `public/images/skill-system-architecture.png`
  - `1580 x 1166`
  - 比例约 `1.36:1`
- `public/images/clipboard-architeture.png`
  - `1334 x 1314`
  - 比例约 `1.02:1`

### 结论

这两张图都不是“特别高、特别难布局”的极端素材。

所以当前问题的核心不是：

- 图片文件分辨率不够
- 图片本身比例异常
- 必须裁图才能成立

而是：

**页面模板还在错误地分配高度。**

---

## 3. 根因判断

当前架构页的问题，不是单一的 `img` 样式问题，而是页面结构问题。

### 根因 A：顶部独立标题区占掉了最宝贵的垂直空间

当前架构页仍然是：

1. 顶部 `SectionHeader`
2. 下方 `body`
3. `body` 内再分左图右栏

这意味着页面先拿出一截高度给标题和 lead，剩余空间才轮到图片。

在一屏展示约束下，这个顺序是错的，因为架构页里最应该优先保证的是图。

### 根因 B：右侧栏虽然变轻了，但仍然在和图片争高度

第二轮把“厚卡片堆叠”改成了 `decision rail`，方向是对的。

但因为顶部标题区还在，所以整个页面仍然是：

- 顶部文本区
- 左图
- 右决策栏

结果就是图片被迫继续缩小。

### 根因 C：当前模板还是“普通内容页思路”，不是“图优先页思路”

架构页本质上不是：

- 先读标题，再读正文，再看配图

而是：

- 先看到图这个主证据
- 再在右侧读解释和工程决策

这两种页面的结构优先级不一样，不能继续共用同一个头部占高逻辑。

---

## 4. 第三轮修正方向

### 新模板名称

推荐为架构页新增一个明确的版式变体：

**`diagramFocus`**

它的核心原则只有一条：

**优先把可用高度留给图。**

### 页面结构

统一改成：

```text
+--------------------------------------------------------------+
| 左侧：架构图（主证据，完整展示） | 右侧：narrative rail      |
|                                  | kicker                    |
|                                  | title                     |
|                                  | lead                      |
|                                  | 01 决策                   |
|                                  | 02 决策                   |
|                                  | 03 决策                   |
|                                  | 04 决策（如有）           |
+--------------------------------------------------------------+
```

关键变化：

1. 取消架构页顶部独立 `SectionHeader`
2. 将 `kicker / title / lead` 收进右侧栏顶部
3. 左侧图占据整页主体高度
4. 右侧只保留必要解释，不再额外制造“上半区说明、下半区决策”的二次切分

---

## 5. 第 7 页与第 10 页的关系

这两页不应该做成两种不同的设计语言。

正确做法是：

**使用同一个 `diagramFocus` 模板，只做参数化调整。**

参数只调两类：

1. 图槽比例
2. 右侧栏密度

### 第 7 页建议

- 图比例：横向，约 `1.36:1`
- 决策数：4 条

建议：

- 左图列更宽
- 右栏更窄，但内部更紧凑
- 4 条决策用编号列表，不用卡片

建议视觉分配：

- 左列约 `64%`
- 右列约 `36%`

### 第 10 页建议

- 图比例：接近正方形，约 `1.02:1`
- 决策数：3 条

建议：

- 左图槽做得更接近正方形
- 右栏可以稍宽一点，但仍保持紧凑

建议视觉分配：

- 左列约 `58%-60%`
- 右列约 `40%-42%`

---

## 6. 明确不要做的事

这轮不建议：

- 不裁图
- 不裁边
- 不修改原图文件尺寸
- 不把图继续缩小去迁就标题区
- 不再回到“左图 + 右侧厚卡片列表”

原因很简单：

当前问题不是素材不行，而是模板没把空间给对地方。

1. 为 `ArchitectureDecisionSection` 增加 `diagramFocus` 变体

2. 第 7 页和第 10 页允许不同的列宽参数
3. rail 内标题与决策之间的间距压缩
4. 决策条目继续使用轻量列表，不改回卡片

### 内容层

1. 第 7 页与第 10 页继续沿用现有图片和现有决策文案
2. 这轮不改图片资源，不改文案主旨

5. 右侧决策仍然完整可读，但视觉优先级低于图片
